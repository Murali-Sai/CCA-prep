// 00 · how to use

The daily rhythm: study → build → drill

Three hours a day, split into three blocks. The exam tests production judgment, not memorization — so you study the concept, build a tiny version of it, then drill exam-style questions on it. That triple loop is what makes it stick.

HOUR 1 — STUDY

Read & understand

Read the linked resource for the day plus the matching section of your two PDFs (Exam Guide + Playbook). Take notes in your own words. If a core AI term is unfamiliar, jump to the Primers first.

HOUR 2 — BUILD

Make a tiny version

Code the day's mini-project. No copy-paste — type it, break it, fix it. Reuse your existing projects (your RAG system, your agent) instead of starting from zero. Push to GitHub daily.

HOUR 3 — DRILL

Test yourself

Do 5–10 exam-style questions on the day's topic (use the free practice bank ↗). For every wrong answer, write down why the right answer wins. Tick the day complete →

Your north-star heuristic for the whole exam: when two answers both look right, pick the one that is explicit, programmatic, and scoped over the one that is implicit, probabilistic, and broad. Hooks beat prompts. Structured errors beat generic ones. Explicit criteria beat "be thorough." This single idea answers ~80% of CCA questions.

Core AI concepts, from scratch

Assume you know nothing. Each card gives you the plain-English idea, how it's actually used in industry, and why it matters for the exam & interviews. Read these first if any term in the roadmap is unfamiliar.

Tap any card to expand. The order builds up: language models → tokens → prompting → embeddings → vector DBs → RAG → tools → MCP → agents → multi-agent → hooks → Claude Code → LLMOps. By the end you'll be able to hold a real conversation about any of them.
🌱 Brand new to AI? Start here first (optional, ~2–3 hrs). You do not need to know any of this yet — and you don't have to finish either course. Skim Microsoft's Generative AI for Beginners (lessons 1–5, ~90 min) for the big picture, and chapter 1 of the Hugging Face LLM Course for how a model actually reads text. The primers below then give you the exam-ready version of each idea — including what it is and how industry uses it. All of these also live in the resource library so you never have to search.
01Large Language Model (LLM) the engine everything sits on▼
In plain English
A model trained on enormous amounts of text to do one deceptively simple thing: predict the next chunk of text (a "token") given everything before it. Do that well enough, billions of times, and you get something that can write, summarize, answer questions, and reason. Claude, GPT, and Gemini are all LLMs.
In industry
LLMs power chatbots, coding copilots, search, document summarization, and customer support. Companies rarely train their own — they call a provider's model (like Claude) through an API and build around it.
Why it matters for CCA
The exam assumes you understand two consequences: an LLM is probabilistic (the same prompt can give different answers — so prompts can't guarantee behavior), and it has a fixed knowledge cutoff and a finite context window. Almost every exam pattern flows from these facts.
Analogy: a brilliant new hire with no memory between conversations and no access to your files — until you give them context and tools.

02Tokens & the Context Window the model's working memory▼
In plain English
Models don't read words — they read tokens (roughly ¾ of a word each). The context window is the maximum number of tokens the model can "see" at once: your instructions, the conversation, any documents, and its own reply all share that budget. Run out, and older content must be dropped or summarized.
In industry
You pay per token (input + output), so token efficiency = cost control. Long PDFs get split into chunks; long chats get summarized. Managing this budget well is a core engineering skill.
Why it matters for CCA
Domain 5 is built on this: the "lost in the middle" effect (models recall the start and end of long context better than the middle), context pruning (trim a 40-field tool response to the 5 fields you need), and "case facts" blocks (pin critical numbers at the top so they're never summarized away).
Anthropic: effective context engineering ↗
03Prompt Engineering controlling output with words▼
In plain English
The craft of writing instructions and examples that get reliable output. Key moves: be explicit about criteria, give few-shot examples (show 2–4 worked examples), ask for step-by-step reasoning, and use XML tags to structure the prompt (Claude responds especially well to these).
In industry
It's the cheapest, fastest lever to improve an AI feature — no retraining, works across model updates. The first thing any AI engineer reaches for before considering anything heavier.
Why it matters for CCA
Domain 4 (20%). The tested rules: explicit measurable criteria beat vague ones ("flag functions over 50 lines" > "flag long functions"); 2–4 few-shot examples is optimal, never more than 6; and vague instructions like "be thorough" are always the wrong answer.
Interview soundbite: "Prompt engineering is iterative spec-writing for a probabilistic system — you define success criteria, then tighten the prompt against failures."
Anthropic prompt engineering docs ↗ DAIR.AI: the complete Prompt Engineering Guide ↗
04Structured Output & tool_use getting clean JSON, reliably▼
In plain English
Apps need predictable data, not prose. By defining a "tool" with a JSON schema and asking the model to call it, you force the reply into an exact shape (fields, types, allowed values). This is the most reliable way to get machine-readable output.
In industry
Every time an LLM feeds a database, an API, or another service, structured output is how the handoff stays clean. Think invoice extraction, form filling, classification pipelines.
Why it matters for CCA
The most-tested nuance: tool_use guarantees structure, not semantics. The JSON will be valid, but the values can still be wrong (wrong vendor name, impossible date). So you always add a validation step afterward. Also know tool_choice: "auto" / "any" / forced specific tool.
Anthropic: prompting best practices (tool use) ↗
05Embeddings turning meaning into numbers▼
In plain English
An embedding turns a piece of text into a long list of numbers (a vector) that captures its meaning. The trick: texts with similar meaning end up as vectors that sit close together in space. "dog" and "puppy" land near each other; "dog" and "spreadsheet" land far apart.
In industry
Embeddings power semantic search (find by meaning, not exact keywords), recommendations, deduplication, and clustering. They're the foundation under RAG.
Why it matters for CCA
Not heavily tested directly (vector DB internals are out of scope), but you must understand them to reason about RAG and about shared vector stores for multi-agent memory, which the Playbook highlights as the right pattern over passing full logs around.
Analogy: a GPS coordinate for meaning — every sentence gets a location, and "nearby" means "similar."
Free guide: embeddings & RAG end-to-end ↗ Hugging Face: how embeddings & tokens work ↗
06Vector Database storage built for similarity▼
In plain English
A database designed to store millions of embeddings and answer one question fast: "which stored vectors are closest to this one?" It uses approximate-nearest-neighbor (ANN) search so similarity lookups stay quick at scale. Popular ones: Chroma, Pinecone, Qdrant, pgvector, FAISS, Milvus.
In industry
It's the retrieval layer of almost every RAG system and semantic-search product. You embed your documents once, store them, then query by similarity at runtime.
Why it matters for CCA
Implementation details are explicitly out of scope, but the architectural idea is in scope: a shared vector store lets subagents write findings and later retrieve only what's relevant — far better than daisy-chaining full conversation logs (which blows up token cost exponentially).
Free guide: vector DB + RAG pipeline ↗
07RAG — Retrieval-Augmented Generation grounding answers in your data▼
In plain English
Before the model answers, you retrieve the most relevant pieces of your own data (using embeddings + a vector DB), paste them into the prompt, and ask the model to answer from those facts. Three steps: Retrieve → Augment → Generate. The model stops guessing and starts citing.
In industry
The dominant pattern for "chat with your docs": company knowledge bases, support bots, legal/medical assistants, research tools. It adds up-to-date and proprietary knowledge without retraining the model, and sharply reduces hallucination.
Why it matters for CCA
RAG concepts underpin the data-extraction scenario, information provenance (keep claim→source mappings through synthesis), and context management. Evaluate it with metrics like faithfulness and context recall (the RAGAS framework).
Interview soundbite: "RAG vs fine-tuning: RAG injects knowledge at query time and is cheap and updatable; fine-tuning bakes it into weights and is expensive and static. Start with RAG."
Free beginner-friendly RAG guide ↗
08Tool Use / Function Calling letting the model take actions▼
In plain English
You hand the model a menu of functions it's allowed to call (each with a name, description, and JSON schema for its inputs). The model decides which to call and fills in the arguments; your code actually runs the function and feeds the result back. That's how a text model can look up an order or send an email.
In industry
The bridge between "the model talks" and "the model does." Every agent that books, queries, or updates anything is using tool calls under the hood.
Why it matters for CCA
Domain 2 (18%). Tested heavily: tool descriptions are how the model picks the right tool (vague descriptions → misrouting); cap an agent at ~4–5 tools (18 degrades selection); and return structured errors (isError, errorCategory, isRetryable) — never a bare empty result for a failure.
Anthropic: tool use guidance ↗
09MCP — Model Context Protocol "USB-C for AI tools"▼
In plain English
An open standard (created by Anthropic) for connecting models to external tools and data. You build an MCP server once that exposes some tools/data; then any MCP client (Claude Desktop, Claude Code, Cursor, etc.) can plug in and use it — no custom integration per app. Three primitives: tools (actions the model calls), resources (data/content catalogs), prompts (templates).
In industry
Rapidly becoming the default way to wire AI into Google Drive, Slack, GitHub, databases, and internal systems. "Build the connector once, reuse everywhere."
Why it matters for CCA
Core to Domain 2 and appears in 4 of 6 scenarios. Know: .mcp.json (project-scoped, shared via git) vs ~/.claude.json (personal); environment-variable expansion for secrets (never hardcode); and resources (expose a catalog) vs tools (perform an action).
Analogy: before USB-C, every device had its own charger. MCP is the universal port — one standard, any tool, any AI client.
modelcontextprotocol.io (official) ↗
10AI Agent & the Agentic Loop an LLM that acts in a loop▼
In plain English
An agent is an LLM running in a loop: it receives a goal, decides whether to call a tool, you run the tool and feed the result back, and it keeps going until the task is done. The loop is driven by stop_reason: if it's "tool_use" the agent wants to act (run the tool, continue); if it's "end_turn" the agent is finished (stop).
In industry
Agents power autonomous coding assistants, customer-support resolution, research automation, and data pipelines — anything that takes multiple steps and decisions rather than one reply.
Why it matters for CCA
Domain 1 is the biggest (27%). The #1 tested anti-pattern: never parse the model's text ("looks done") to decide when to stop — always check stop_reason. Tool results get appended to history so the model can reason about the next step.
Anthropic: building effective agents ↗
11Multi-Agent Systems (Hub-and-Spoke) a coordinator + specialists▼
In plain English
For complex jobs, one coordinator agent breaks the work into subtasks and delegates each to a specialized subagent (a web-searcher, a doc-analyzer, a synthesizer). The coordinator routes all communication — that's the "hub-and-spoke" shape. Crucially, subagents start with isolated context — they don't automatically see the coordinator's conversation, so you must pass what they need explicitly.
In industry
Research pipelines, large-scale code review, and any workflow too big for one context window. Specialization keeps each agent focused and its tool list small.
Why it matters for CCA
Heavily tested in Domain 1. Know: spawn subagents in parallel by emitting multiple Task calls in one coordinator response; the coordinator's allowedTools must include Task; and the classic failure — too-narrow task decomposition leaving whole topics uncovered (the root cause is the coordinator's plan, not the subagents).
CCA: Agentic Architecture domain ↗
12Hooks & Deterministic Guardrails guarantees, not suggestions▼
In plain English
A hook is your own code that runs automatically at a fixed point in the agent's lifecycle. PreToolUse fires before a tool runs (you can block or modify it); PostToolUse fires after (you can normalize or log the result). Because it's plain code, it's deterministic — it runs 100% of the time, unlike a prompt instruction the model might ignore.
In industry
Used for compliance ("never refund over $500"), audit logging, and cleaning up messy tool outputs (e.g., normalizing different date formats) before the model sees them.
Why it matters for CCA
This is the recurring exam theme: when a question says "guaranteed," "deterministic," or "zero failure," the answer is a hook or programmatic prerequisite — never a system prompt, few-shot, or classifier. Prompts have a non-zero failure rate; hooks don't.
Anthropic: Agent SDK hooks ↗
13Claude Code & the Agent SDK the coding agent + its config▼
In plain English
Claude Code is Anthropic's agentic coding tool (terminal/IDE). You configure it with a CLAUDE.md file (project instructions), skills (reusable multi-step behaviors), slash commands, path rules, and MCP servers. The Agent SDK is the same engine, programmable in Python/TypeScript, so you can build your own agents on it.
In industry
Dev-productivity, automated code review in CI/CD, refactoring, test generation, and exploring unfamiliar codebases. Run it non-interactively with the -p flag inside pipelines.
Why it matters for CCA
Domain 3 (20%). Know the CLAUDE.md hierarchy (user → project → directory), commands vs skills (skills can run isolated via context: fork), plan mode vs direct execution, path-scoped .claude/rules/, and the -p / --output-format json CI flags.
Anthropic: Agent SDK overview ↗
14LLMOps (Tracing, Evals, Versioning) keeping AI reliable in prod▼
In plain English
The "operations" layer for LLM apps: tracing every run so you can see what happened, versioning prompts, building eval suites that score quality automatically, and monitoring cost/latency. Tools: LangSmith, Langfuse.
In industry
It's what separates a demo from a production system. When an agent misbehaves, traces tell you exactly which tool call or prompt went wrong.
Why it matters for CCA
Not a named domain, but it reinforces Domain 5 (reliability) and is a strong interview topic. Bonus: every failure mode you see in a trace maps to a CCA anti-pattern (looping → stop_reason; bad tool pick → tool descriptions; context overflow → pruning).
Anthropic engineering blog ↗
15Batch API & Cost/Latency Trade-offs cheaper when you can wait▼
In plain English
For work that isn't time-sensitive, the Message Batches API processes large jobs at ~50% lower cost with up to a 24-hour turnaround and no latency guarantee. You correlate each request/response with a custom_id.
In industry
Overnight report generation, weekly audits, bulk document processing — anything that can run while no one is waiting.
Why it matters for CCA
A favorite Domain 4 question: don't route a blocking workflow (a pre-merge check developers wait on) through the batch API — "often faster than 24h" is never an SLA. Real-time for blocking; batch for overnight. Also: batch can't do multi-turn tool calling mid-request.
Anthropic: Message Batches API ↗
16Prompting vs RAG vs Fine-tuning the "which lever?" question▼
In plain English
Three ways to make a model better at your task. Prompting: change the instructions — instant, free, first thing to try. RAG: feed in relevant facts at query time — best when you need current or proprietary knowledge. Fine-tuning: retrain the weights on your data — expensive, static, for when you need a specific style/behavior at scale.
In industry
The standard decision ladder: try prompting → add RAG for knowledge → only fine-tune if the first two can't hit the bar. Most production systems never need fine-tuning.
Why it matters for CCA
Fine-tuning is explicitly out of scope for the exam — but knowing why (the exam favors prompting/RAG/architecture solutions over retraining) helps you eliminate distractors that propose training a model when a simpler lever exists.
Interview soundbite: "Prompting changes what you say, RAG changes what the model sees, fine-tuning changes what the model is. Reach for them in that order."

// 04 · interview-ready talking points

Talk about any of this confidently

Short, sharp answers to the questions you'll actually get asked. Each one doubles as a CCA concept check.

"What's the difference between RAG and fine-tuning?"
RAG injects relevant facts at query time — cheap, updatable, great for proprietary/current knowledge. Fine-tuning bakes behavior into the weights — expensive, static, for consistent style at scale. Start with prompting, add RAG, fine-tune last.
"How do you make an agent reliable in production?"
Enforce critical rules with hooks/programmatic gates, not prompts (prompts have a non-zero failure rate). Return structured errors so the agent can recover. Trace everything (LLMOps). Use an independent reviewer instance, never same-session self-review.
"When would you use a multi-agent system?"
When the task is too big for one context window or needs specialization. A coordinator delegates to subagents with isolated context and small tool sets. Watch for the failure mode: too-narrow decomposition leaving topics uncovered.
"What is MCP and why does it matter?"
An open standard — "USB-C for AI" — to connect models to tools/data. Build a server once; any client can use it. It replaces fragmented per-app integrations with one protocol. Three primitives: tools, resources, prompts.
"How do you get reliable structured output?"
Define a tool with a JSON schema and force it with tool_choice. That guarantees structure — but not semantics, so I add a validation step (e.g., totals must reconcile) and use nullable fields to stop the model fabricating missing data.
"How do you manage long conversations?"
Pin critical facts in an immutable "case facts" block at the top (high-recall position), prune verbose tool outputs to needed fields, summarize resolved history but keep the active issue verbatim, and use scratchpad files to survive context limits.
// 05 · deploy to vercel

Put this hub online in ~3 minutes

This is a static site — no build step. Pick the easiest path. The whole folder (this index.html + the resources/ PDFs + vercel.json) deploys as-is.

Option A — Drag & drop (no account setup, fastest)

Unzip the project

Download the bundle I gave you and unzip it. You'll see index.html, a resources/ folder, vercel.json, and README.md.

Go to Vercel

Open vercel.com/new and sign in (free — use your GitHub or email).

Drag the folder in

On the import screen, drag your unzipped folder onto the upload area. Vercel auto-detects a static site. Click Deploy. You'll get a live https://your-name.vercel.app URL.

Option B — GitHub (best for editing later)

Create a repo & push

# inside the unzipped folder
git init
git add .
git commit -m "CCA speedrun study hub"
git branch -M main
git remote add origin https://github.com/<you>/cca-speedrun.git
git push -u origin main

Import to Vercel

At vercel.com/new, pick Import Git Repository, select your repo, framework preset Other, leave build settings empty, and Deploy. Every future git push auto-redeploys.

Option C — Vercel CLI (one command)

Install & deploy

npm i -g vercel
cd cca-speedrun        # your unzipped folder
vercel                 # follow prompts → preview URL
vercel --prod          # promote to production
Note on the bundled PDFs: the Architect's Playbook is ~14 MB, which is fine for Vercel. If you'd rather keep the deploy tiny, delete the resources/ folder before deploying and just keep the PDFs locally — the rest of the site works unchanged (those two links will simply 404).
